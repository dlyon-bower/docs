title: "Workspace memory"
description: "Standing instructions for Bird that apply to every conversation in this workspace — vocabulary, conventions, and team conventions."

Workspace memory is a short note shared across your lab that tells Bird how *this* workspace works. It's read at the start of every conversation in the workspace and applies to every member.

If personal context is "this is me", workspace memory is "this is how we do things here".

## Why use it

Labs have conventions that aren't obvious to a general-purpose assistant. Citation style. Sample naming schemes. The fact that "subjects" is preferred over "patients". Whether Bird should suggest external collaborators when the user asks for collaborators.

Without workspace memory, every member ends up repeating those rules in their own conversations. With it, Bird knows them once and applies them for everyone.

The benefits across a team:

<AccordionGroup>
  <Accordion title="Shared vocabulary">
    Lab-specific terminology, naming conventions, and method shorthand are encoded once. Bird uses your team's terms instead of generic ones.
  </Accordion>
  <Accordion title="Standing instructions for everyone">
    "Always include a Methods bullet when summarising a paper." "When proposing collaborators, only suggest current lab members." "Default reference genome is mm10 unless stated otherwise." Set once, applied across the team.
  </Accordion>
  <Accordion title="Onboarding new members">
    A new postdoc joins the workspace and Bird already knows your conventions on day one. They don't have to discover them by getting things wrong first.
  </Accordion>
  <Accordion title="Recurring entities">
    People, active projects, key methods, instruments that come up often — Bird has them in context without anyone having to explain.
  </Accordion>
  <Accordion title="Negative-space rules">
    "Don't invent citations." "Don't recommend external collaborators without explicit ask." Telling Bird what *not* to do is often more powerful than telling it what to do.
  </Accordion>
</AccordionGroup>

## How to write it

Free text. Write it like a lab handbook entry, not a rules document. Bird reads it as natural language.

A useful starting structure:

- **About this workspace** — one or two sentences on what the lab does
- **Vocabulary and conventions** — terms, shorthands, naming schemes
- **Standing instructions** — behaviour Bird should always follow
- **Recurring entities** — people, active projects, key methods, instruments
- **Don'ts** — things that should never happen

### Example

```
We're a marine biology lab at the University of Bristol focused on coral
alkalinity tolerance. Field samples are collected weekly from three reef sites
and processed within 24 hours.

Vocabulary:
- We call participating reef sites "stations", not "sites". Stations are
  numbered S1, S2, S3.
- "Batch" refers to a weekly collection cycle, not an individual sample.

Standing instructions:
- Citation style is AMA. No footnotes.
- When summarising a paper, always include a Methods bullet.
- When proposing collaborators, only suggest current lab members listed in
  the People section below.

People:
- Dr. Lin (PI), Sara (postdoc), Tom (PhD), Alex (PhD)

Active projects:
- Project Atlas — coral alkalinity tolerance
- Project Reef-Watch — site monitoring infrastructure

Don't:
- Don't recommend external collaborators without an explicit ask.
- Don't invent citations or fabricate sources.
```

## Where to set it

**Settings → Workspace → Workspace memory for Bird**

Workspace memory is **edited only by workspace owners and admins**. Members and viewers see it in read-only mode so they understand what's shaping Bird's behaviour, but can't change it.

Changes apply to new conversations across the whole workspace — Bird reads memory at the start of each conversation, so existing chats keep their original context until reopened.

The 12 KB cap fits a substantial set of conventions. If you find yourself near the limit, prune older items rather than expanding.

## HIPAA workspaces

If your workspace has HIPAA mode enabled, Bird automatically adds a locked compliance section to its system instructions. Workspace memory must not contain:

- Patient identifiers
- Dates of birth
- Medical record numbers
- Phone numbers in a structured pattern

Saves are rejected if any of those patterns are detected. The DB and audit trail never store memory content for non-HIPAA workspaces either, but the rejection on HIPAA workspaces is an explicit defence-in-depth.

<Note>
  Workspace memory is for **workspace structure and conventions only** — never for sensitive subject data. Patient records and study data live in encrypted notes and are referenced by ID, not by name in memory.
</Note>

## Privacy and audit

Workspace memory is shared with everyone in the workspace, so write with that in mind. Don't put anything there that you wouldn't want every member to read.

For accountability:

- Every save is recorded in the workspace audit log — who edited it, when, and how many bytes were added or removed
- The actual content of past versions is **not** stored in audit, so admins can't recover historical content from there
- Audit visibility is limited to what's needed for accountability without leaking removed instructions back to members

## Workspace vs. personal memory

Two layers, two scopes:

|  | Workspace memory | [Personal context](/ai-and-search/personal-context) |
| --- | --- | --- |
| Visible to | All workspace members | Only the user |
| Edited by | Owners and admins | The user themselves |
| Use it for | Lab vocabulary, conventions, standing instructions | Your role, preferences, hard rules |
| Applies to | Every conversation in the workspace | Every conversation where you're the requester |

Both are loaded into Bird's context simultaneously. Workspace rules win on compliance and shared conventions; personal preferences win on individual style.

## Tips

<Tip>
  **Treat it like a living lab handbook.** When the lab adopts a new convention or retires an old one, update workspace memory the same week. Stale conventions are worse than no conventions.
</Tip>

<Tip>
  **Negative-space rules are powerful.** "Don't recommend external collaborators" is often more useful than "Recommend internal collaborators only" — it gives Bird a clear boundary without prescribing every alternative.
</Tip>

<Tip>
  **Review with the team quarterly.** Workspace memory is shared infrastructure. A short read-through at lab meetings catches stale items and surfaces new conventions worth adding.
</Tip>
